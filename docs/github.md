# 카카오임팩트 GitHub 사용 가이드

작성일: 2026년 6월 16일 · 최종 수정: 2026년 6월 19일

재단이 만든 코드와 웹 결과물은 **GitHub**에서 함께 관리해요. 누가 만들었든 재단의 공용 자산으로 남고, 안전하게 세상에 내보낼 수 있도록 아래 3가지 원칙을 지켜주세요.

> **원칙 1**: 재단 코드는 개인 계정이 아니라 **재단 공식 Organization**에 둡니다.
>
> **원칙 2**: 모든 레포는 **비공개(Private)가 기본**, 외부 공개는 검토를 거칩니다.
>
> **원칙 3**: 웹에 올릴 때는 **접근 제한 · 색인 차단 · 비용 통제**를 함께 챙깁니다.

## 🚀 빠른 안내: 이럴 땐 이렇게

> 아래만 읽어도 대부분 해결돼요. 더 자세히 알고 싶을 때만 해당 번호 섹션으로 내려가세요.

**Q. 내가 만든 코드나 사이트, 어디에 올려요?**
→ 재단 GitHub 조직(`kakao-impact-foundation`)에 **비공개로** 올려요. 개인 계정에 두지 마세요. *(자세히 → 1·2번)*

**Q. 그냥 구글 드라이브에 두면 안 돼요? GitHub 왜 써요?**
→ 문서는 드라이브, **코드는 GitHub**예요. 버전 기록·함께 고치기·웹 배포가 GitHub에서 훨씬 수월해요. *(→ 1번)*

**Q. 만든 사이트를 웹에 올리고 싶은데, 아무나 보면 안 돼요.**
→ 세 가지를 같이 거세요. ① 비밀번호 게이트 ② 검색 차단(`noindex`) ③ (AI API 쓰면) 비용 한도. 진짜 가려야 할 데이터면 **서버에서 검증**해야 해요. *(→ 4번)*

**Q. 만든 사이트를 웹에 띄우고 싶어요. (GitHub Pages)**
→ 재단 Org 레포를 **공개로 전환하면** GitHub Pages로 바로 띄울 수 있어요. 주소가 `kakao-impact-foundation.github.io/레포이름/`으로 나와서 **재단 공식 사이트처럼 보입니다.** 단 공개 레포라 소스가 다 보이니, 민감정보·비밀번호는 빼고 검색 차단(`noindex`)을 함께 거세요. *(→ 3번)*

**Q. 외부에 공개해도 돼요?**
→ 공개 전에 **테크임팩트팀(테일러)에 먼저 알려주세요.** 민감정보가 없는지 함께 확인하고 공개해요. 한 번 공개하면 되돌리기 어려우니 신중하게. *(→ 2번)*

**Q. Claude 같은 AI API를 쓰는 봇·도구를 만들었어요.**
→ 배포 전에 **꼭 알려주세요.** 비용이 폭주하지 않게 월 한도·알림부터 걸어요. *(→ 4-3번)*

**Q. 내 개인 계정에 이미 재단 코드가 있어요. 어떻게 옮겨요?**
→ 레포 `Settings` → `Danger Zone` → `Transfer`로 조직에 옮기면 끝. 이슈·이력·주소 연결이 다 따라와요. *(→ 2번)*

**Q. 만든 페이지를 외부엔 안 보이고 카임 사람들만 보게 하고 싶어요.**
→ Apps Script 웹앱으로 배포할 때 **액세스 권한을 "Kakao Impact 내의 모든 사용자"로** 설정하면 돼요. 재단 구글 계정 로그인한 사람만 보여요. 무료·10분 (소울 가이드 있음). *(→ 5번)*

**Q. 내가 만든 에이전트·스킬을 다른 크루랑 나누고 싶어요.**
→ 공유 레포 `ax-harness` 한 곳에 모아요. **조직 안에 새 레포를 따로 만들지 마세요.** *(→ 7번)*

## 1. 재단 GitHub은 어떻게 생겼나요

GitHub에는 **개인 계정**(User)과 **조직 계정**(Organization) 두 종류가 있어요. 재단은 조직 계정을 따로 만들어 두었고, 크루는 각자 개인 계정으로 여기에 멤버로 참여합니다.

| 구분 | 계정 | 용도 |
| :-- | :-- | :-- |
| 공식 Org | `kakao-impact-foundation` | 모든 재단 레포가 사는 곳 |
| 관리자 계정 | `kakaoimpact-admin` | 결제·복구 같은 비상용. 일상 작업은 안 함 |
| 크루 | 각자 개인 GitHub 계정 | Org 멤버로 초대받아 참여 |

**이런 건 피해주세요**

- 계정 하나를 여러 명이 같이 쓰기: 2FA를 공유할 수 없고 누가 무엇을 했는지 추적이 안 됩니다.
- 회사 메일로 크루마다 새 GitHub 계정 만들기: 도메인을 잃으면 복구가 어려워요. GitHub은 1인 1계정이 원칙입니다.
- 재단 코드를 개인 계정에 그대로 두기: 위의 이유로 꼭 Org로 옮겨주세요.

## 2. 레포는 비공개가 기본입니다

| 종류 | 누가 보나요 | 기본 |
| :-- | :-- | :-- |
| **Private (비공개)** | Org 멤버 중 권한 받은 사람만 | ✅ 기본값 |
| **Public (공개)** | 전 세계 누구나 | 검토 후 결정 |

> 외부 공개는 되돌리기 어려운 결정이에요. 한 번 공개된 코드는 누군가 이미 복제했을 수 있으니, **공개 전 민감정보 점검**을 꼭 거칩니다.

**공개로 전환하고 싶다면** 테크임팩트팀(테일러)에 알려주세요. API 키·개인정보·내부 데이터가 섞여 있지 않은지 함께 확인한 뒤 공개합니다.

**내 코드를 재단 Org로 옮기려면 (Transfer)**

레포 → `Settings` → 맨 아래 `Danger Zone` → `Transfer ownership` → 새 소유자에 `kakao-impact-foundation` 입력. 이슈·히스토리·링크가 모두 따라오고 옛 주소는 자동으로 새 주소로 연결돼요. 옮긴 뒤 공개 범위(Private/Public)만 한 번 더 확인해주세요.

## 3. GitHub Pages로 사이트 배포하기

GitHub Pages는 레포 안의 정적 파일(HTML·CSS·JS)을 **무료로 웹에 띄워주는** 기능이에요. 별도 서버 없이 가이드·대시보드·소개 페이지 같은 걸 바로 공개할 수 있습니다.

### ★ 재단 Org에 두고 띄우면 주소부터 재단 것이 됩니다

같은 사이트라도 **어느 계정에서 띄우느냐에 따라 주소가 달라져요.**

| 띄운 곳 | 주소 모양 | 인상 |
| :-- | :-- | :-- |
| 개인 계정 | `내아이디.github.io/레포이름/` | 개인 작업물처럼 보임 |
| **재단 Org** | **`kakao-impact-foundation.github.io/레포이름/`** | **재단 공식 자산처럼 보임** ✅ |

예를 들어 이 가이드 사이트도 재단 Org에서 띄워서 주소가 이렇게 나와요.

```
https://kakao-impact-foundation.github.io/ax-guides/viewer.html?doc=github
```

주소 첫머리에 `kakao-impact-foundation`이 박혀 있으니, 외부에 공유해도 한눈에 "재단이 만든 것"으로 읽힙니다. **그래서 재단에서 만든 사이트는 개인 계정이 아니라 재단 Org에 레포를 두고 Pages로 배포하는 걸 권장해요.** 개인 계정에 띄워 둔 사이트가 있다면, 2번의 Transfer로 Org에 옮긴 뒤 다시 띄우면 주소가 재단 것으로 바뀝니다.

### 띄우는 법 (무료 플랜 기준)

무료 플랜에서는 **공개(Public) 레포만** Pages를 띄울 수 있어요. 그래서 순서는 이래요.

1. 레포를 공개로 전환 (2번의 민감정보 점검을 먼저 거치고)
2. 레포 `Settings` → `Pages` → Branch를 `main` / 폴더 `/`로 지정 → Save
3. 잠시 뒤 `kakao-impact-foundation.github.io/레포이름/` 으로 접속 확인

### 공개 레포라서 꼭 챙길 것

Pages로 띄우려면 레포가 공개여야 하니, **소스 코드가 전부 외부에 보입니다.** 아래를 반드시 지켜주세요.

- **API 키·비밀번호·토큰을 코드에 넣지 마세요.** 공개 레포에선 그대로 노출돼요.
- **화면에 그려지는 데이터도 공개**입니다. 개인정보·내부 수치를 화면에 직접 띄우면 안 돼요.
- **검색 색인 차단(`noindex` + `robots.txt`)을 꼭 거세요.** 주소를 모르면 못 찾게 (→ 4-2번).
- 화면 앞 **비밀번호(클라이언트)는 공개 레포에선 소스를 열면 보여서** "지나가다 못 들어오는" 약한 차단일 뿐이에요. 진짜 가려야 하면 Pages 말고 **Vercel 서버 검증**이나 **Apps Script 카임 SSO**를 쓰세요 (→ 5번).

## 4. 안전하게 배포하기: 세 가지를 함께

웹에 무언가 올릴 때는 아래 세 개를 **하나만 믿지 말고 같이** 겁니다.

### 4-1. 접근 제한: 누가 볼 수 있나

가려야 할 데이터가 있느냐에 따라 보호 강도가 달라요.

| 레벨 | 방식 | 강도 | 쓸 곳 |
| :-- | :-- | :-- | :-- |
| 캐주얼 차단 | 화면 앞 비밀번호(클라이언트) + 검색 차단 | 약함 (공개 레포면 비번 노출) | 민감하진 않지만 검색에 안 걸리게 |
| 실질 인증 | 서버에서 검증(Vercel + 쿠키 + 환경변수) | 강함 | 내부 데이터·업무 도구 |

> 정말 가려야 할 데이터가 있으면 **반드시 서버에서 검증**하세요. 화면 앞 비밀번호만으로는 부족합니다 (단, 3번처럼 레포가 비공개면 캐주얼 차단도 한결 단단해져요).

### 4-2. 검색 색인 차단: 접근 제한과 별개로 항상

검색엔진에 안 걸리게 막습니다. 정적 사이트면 `<head>`에 아래 한 줄을 넣어주세요.

```html
<meta name="robots" content="noindex, nofollow">
```

`robots.txt`로 `Disallow: /`를, 서버가 있으면 `X-Robots-Tag: noindex` 헤더도 함께 거는 게 좋아요.

### 4-3. API 비용 통제: AI API를 쓰는 도구라면 필수

Claude 같은 API를 쓰는 도구는 잘못하면 비용이 폭주할 수 있어요. **관리자가 콘솔에서 월 사용 한도와 알림을 먼저 걸어두는 게 마지막 방어선**입니다. 앱에서는 요청 횟수 제한(rate limit), 프롬프트 캐싱, 가벼운 작업은 더 저렴한 모델로 분리하는 식으로 아낍니다.

> 비용 통제는 ax-admin·테크임팩트팀이 함께 설정해요. AI API를 쓰는 도구를 만들면 배포 전에 꼭 알려주세요.

## 5. 어디에 올릴까요: 플랫폼 선택

| 상황 | 권장 | 비용 |
| :-- | :-- | :-- |
| 정적 사이트 (소스 공개 OK·가벼운 차단) | **재단 Org 공개 레포 + GitHub Pages** | 무료 |
| API·빌드가 필요한 앱 / 실질 인증 필요 | **Vercel** + 서버 비밀번호 게이트 | 무료로 충분 |
| 사내용 페이지(대시보드·리포트·안내) | **Apps Script 웹앱, 재단 구글 계정 로그인한 사람만** (외부 차단, 무료, 10분) | 무료 |
| 신분증·통장·개인 정산 등 진짜 민감 데이터 | 위 SSO 페이지에도 띄우지 말고, 드라이브 가이드의 개인정보 처리 절차(zip 잠금 → 더존 전자결재) | — |

> **💡 "외부엔 안 보이고 카임 사람들만" 보게 하는 두 가지.** 둘 중 **하나만 고르면 돼요.**
> ① **Apps Script 카임 SSO** (소울 가이드): 웹앱 배포 시 액세스 권한을 "Kakao Impact 내의 모든 사용자"로 두면, 재단 구글 계정으로 로그인한 사람만 봐요. 셋업이 조금 손이 가지만 **누가 보는지 가장 명확합니다.** → [슬랙 안내](https://kakao-impact.slack.com/archives/C0ALY4BSAD6/p1776823766539729)
> ② **비밀번호 게이트**: 비번 아는 사람만. 외부 파트너와도 나눌 수 있지만 비번을 따로 관리해야 해요.
> 더 명확한 건 **① 소울 방식**이에요. 단, SSO는 카임 구성원이면 누구나 보니, 특정 담당자만 봐야 하면 접근을 더 좁히거나 로컬에 두세요.

> 정적 사이트는 GitHub Pages(재단 Org 공개 레포), 서버 검증·API가 필요한 앱은 Vercel, 카임 내부 전용 페이지는 Apps Script SSO — 이 세 가지로 거의 다 해결돼요.

## 6. 배포 전 체크리스트

- [ ] 레포에 API 키·비밀번호가 들어가 있지 않나요 (`.env`, `*.key` 등은 `.gitignore`)
- [ ] 내부용이면 접근 제한(서버 비밀번호 게이트)이 실제로 동작하나요
- [ ] 검색 색인 차단(`noindex`)을 걸었나요
- [ ] AI API를 쓴다면 월 한도·알림·요청 제한을 걸었나요
- [ ] 화면에 개인정보·내부 데이터가 그대로 노출되지 않나요
- [ ] README에 공개 범위와 민감도를 적어두었나요
- [ ] 배포 후 실제 URL에서 차단·동작을 눈으로 확인했나요

## 7. 만든 에이전트·스킬은 ax-harness 한 곳에 모아요

각자 만든 Claude 에이전트·스킬은 흩어 두지 말고 공유 레포 한 곳에 모읍니다.
→ **`kakao-impact-foundation/ax-harness`** (조직 안 비공개 레포)

> ⚠️ **조직 안에 비슷한 레포를 따로 만들지 마세요.** 새 레포를 만들면 공유 카탈로그에 안 잡히고, 결국 "ax-harness 안으로 옮기고 원래 레포는 지워주세요" 하고 다시 손이 가요. 처음부터 ax-harness 안에 올리면 됩니다.

### 어디에 올리나요 (세 칸 중 하나)

| 칸 | 올릴 것 | 경로 |
| :-- | :-- | :-- |
| `shared/` | 누구나 그대로 가져다 쓸 수 있는 **범용** 스킬·에이전트 | `ax-harness/shared/` |
| `team/{팀}/` | 우리 팀이 함께 쓰는 하네스 묶음 | `ax-harness/team/{팀이름}/` |
| `krew/{이름}/` | 내 개인 갤러리 (남이 보고 배우는 용) | `ax-harness/krew/{내영문이름}/` |

지금 `shared`에 스킬 두 개가 올라와 있어요. 누구나 재사용할 만한 범용 도구면 shared 경로에 맞춰 올려주세요. → [shared 폴더](https://github.com/kakao-impact-foundation/ax-harness/tree/main/shared)

### 올리는 법

- GitHub 웹에서 해당 폴더를 열고 `Add file` → `Upload files`로 바로 올려도 돼요 (터미널 없이).
- 올리면 GitHub Actions가 README의 카탈로그를 **자동으로 갱신**해요.
- 각 파일 맨 위 frontmatter(`name`·`description`·`author` 등)만 채우면 카탈로그에 깔끔하게 잡혀요.
- 공유 전에 원온원·예산·개인정보 같은 내부 맥락은 빼주세요. Org 멤버 전원이 보니까요.

## 부록: 관리자(Org Owner)용

- **보안 기본값** (멤버 초대 전 잠금, ✅ 적용 완료): 2FA 강제 · 기본 권한 None/Read · 신규 레포 Private only · Owner 2명 이상(Bus factor 대응)
- **신규 멤버 초대**: `Org → People → Invite member`, 기본 역할은 Member. Owner는 신중하게
- **관리자 계정** `kakaoimpact-admin`은 결제·복구 같은 비상용으로만. 일상 작업은 개인 계정으로
- **퇴사 프로세스**: 본인 소유 레포를 Org로 Transfer → Org 멤버에서 remove → 회사 메일 정리
- 상세 운영 절차·트러블슈팅은 ax-publisher 에이전트와 `github-org-setup.md` 참조
