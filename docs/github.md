# 카카오임팩트 GitHub 사용 가이드

작성일: 2026년 6월 16일 · 최종 수정: 2026년 6월 19일

재단이 만든 코드와 웹 결과물은 **GitHub**에서 함께 관리해요. 재단의 공용 자산으로 남고, 안전하게 세상에 내보낼 수 있도록 아래 3가지 원칙을 지켜주세요.

> **원칙 1**: 재단 코드는 개인 계정이 아니라 **재단 공식 Organization**에 둡니다.
>
> **원칙 2**: 모든 레포는 **비공개(Private)가 기본**, 외부 공개는 검토를 거칩니다.
>
> **원칙 3**: 웹에 올릴 때는 **이 사이트가 외부 공개용인지 내부 공유용인지부터 구분**하고, 종류에 맞는 보호를 챙깁니다.

## 🚀 빠른 안내: 이럴 땐 이렇게

> 아래만 읽어도 대부분 해결돼요. 더 자세히 알고 싶을 때만 해당 번호 섹션으로 내려가세요.

**Q. 내가 만든 코드나 사이트, 어디에 올려요?**
→ 재단 GitHub 조직(`kakao-impact-foundation`)에 **비공개로** 올려요. *(자세히 → 1·2번)*

**Q. 그냥 구글 드라이브에 두면 안 돼요? GitHub 왜 써요?**
→ 문서는 드라이브, **코드는 GitHub**예요. 버전 기록·함께 고치기·웹 배포가 GitHub에서 훨씬 수월해요. *(→ 1번)*

**Q. 만든 사이트를 웹에 올리고 싶은데, 아무나 보면 안 돼요.**
→ 세 가지의 보호 조치가 필요해요. ① 비밀번호 게이트 ② 검색 차단(`noindex`) ③ (AI API 쓰면) 비용 한도. 민감정보 데이터면 **서버에서 우선 검증**해야 해요. *(→ 4번)*

**Q. 만든 사이트를 웹에 띄우고 싶어요. (GitHub Pages)**
→ 재단 Org 레포를 **공개로 전환하면** GitHub Pages를 이용하여 웹으로 바로 띄울 수 있어요. 주소가 `kakao-impact-foundation.github.io/레포이름/`으로 나와서 **재단 공식 사이트처럼 보입니다.** 단 공개 레포라 소스가 다 보이니 민감정보·비밀번호는 꼭 빼세요. 검색 차단(`noindex`)·비번 게이트는 **사이트 성격에 따라 달라요** — 외부에 알리려고 만든 사이트면 오히려 검색이 잘 돼야 하고, 내부용이면 둘 다 보호 조치가 필요해요. *(→ 3번)*

**Q. 사이트는 외부에 보여줘야 하는데, 소스 코드(레포)는 가리고 싶어요.**
→ 레포 비공개 + 외부 공개는 GitHub Pages 무료로는 안 돼요. **Vercel Pro**가 필요한데 테일러가 계정을 갖고 있으니 **테크임팩트팀(테일러)에 문의**하세요. *(→ 3·5번)*

**Q. 외부에 공개해도 돼요?**
→ 공개 전에 **테크임팩트팀(테일러)에 먼저 알려주세요.** 민감정보가 없는지 함께 확인하고 공개해요. *(→ 2번)*

**Q. Claude 같은 AI API를 쓰는 봇·도구를 만들었어요.**
→ AI 기능을 활용한 경우라면, 배포 전에 **꼭 알려주세요.** 비용이 폭주하지 않게 월 한도·알림부터 걸어요. *(→ 4-3번)*

**Q. 내 개인 계정에 이미 업무 관련된 코드가 있어요. 어떻게 옮겨요?**
→ 레포 `Settings` → `Danger Zone` → `Transfer`로 조직에 옮기면 끝. 이슈·이력·주소 연결이 다 따라와요. *(→ 2번)*

**Q. 만든 페이지를 외부엔 안 보이고 내부에서만 보게 하고 싶어요.**
→ 가이드·리포트·안내처럼 화면이 정적인 문서면 **클로드 아티팩트로 발행**하는 게 제일 간단해요. 배포 셋업 없이 바로 뜨고, 재단 claude.ai 조직 멤버만 보여요(외부 공개 설정 자체가 없어요). 서버·API가 붙는 앱이거나 꼭 사이트로 올려야 하면 GitHub Pages에 **비밀번호 게이트**를 거세요. *(→ 5번)*

**Q. 내가 만든 에이전트·스킬을 다른 크루랑 나누고 싶어요.**
→ 공유 레포 `ax-harness` 한 곳에 모아요. **조직 안에 새 레포를 따로 만들지 마세요.** *(→ 7번)*

## 1. 재단 GitHub은 어떻게 생겼나요

GitHub에는 **개인 계정**(User)과 **조직 계정**(Organization) 두 종류가 있어요. 재단은 조직 계정을 따로 만들어 두었고, 크루는 각자 개인 계정으로 여기에 멤버로 참여합니다.

| 구분 | 계정 | 용도 |
| :-- | :-- | :-- |
| 공식 Org | `kakao-impact-foundation` | 재단에서 만든 결과물을 모으고 관리하는 곳 |
| 관리자 계정 | `kakaoimpact-admin` | 결제·복구 같은 비상용 어드민 계정 |
| 크루 | 각자 개인 GitHub 계정 | Org 멤버로 초대받아 참여 |

**이런 건 피해주세요**

- 계정 하나를 여러 명이 같이 쓰기: 2FA(이차인증)를 공유할 수 없고 누가 무엇을 했는지 추적이 안 됩니다. GitHub은 1인 1계정이 원칙입니다.
- 재단 업무를 위해 만든 코드를 개인 계정에 그대로 두기: 향후 관리나 서로 공유, 웹 공개 시 도메인 이슈 등을 위해 Org로 옮겨주세요.

## 2. 레포는 비공개가 기본입니다

| 종류 | 누가 보나요 | 기본 |
| :-- | :-- | :-- |
| **Private (비공개)** | Org 멤버 중 권한 받은 사람만 | ✅ 기본값 |
| **Public (공개)** | 전 세계 누구나 | 검토 후 결정 |

> 외부 공개는 되돌리기 어려운 결정이에요. 한 번 공개된 코드는 누군가 이미 복제했을 수 있으니, **공개 전 민감정보 점검**을 꼭 거치는 것이 좋아요.

**공개로 전환하고 싶다면** 테크임팩트팀(테일러)에 알려주세요. API 키·개인정보·내부 데이터가 섞여 있지 않은지 함께 확인한 뒤 공개합니다.

**내 코드를 재단 Org로 옮기려면 (Transfer)**

레포 → `Settings` → 맨 아래 `Danger Zone` → `Transfer ownership` → 새 소유자에 `kakao-impact-foundation` 입력. 이슈·히스토리·링크가 모두 따라오고 옛 주소는 자동으로 새 주소로 연결돼요. 옮긴 뒤 공개 범위(Private/Public)만 한 번 더 확인해주세요.

## 3. GitHub Pages로 사이트 배포하기

GitHub Pages는 레포 안의 정적 파일(HTML·CSS·JS)을 **무료로 웹에 띄워주는** 기능이에요. 별도 서버 없이 가이드·대시보드·소개 페이지 같은 홈페이지라면, 바로 공개할 수 있습니다.

### ★ 재단 Org에 두고 띄우면 도메인을 연결할 수 있어요.

같은 사이트라도 **어느 계정에서 띄우느냐에 따라 주소가 달라져요.**

| 띄운 곳 | 주소 모양 | 인상 |
| :-- | :-- | :-- |
| 개인 계정 | `내아이디.github.io/레포이름/` | 개인 작업물처럼 보임 |
| **재단 Org** | **`kakao-impact-foundation.github.io/레포이름/`** | **재단 공식 자산처럼 보임** ✅ |

예를 들어 이 가이드 사이트도 재단 Org에서 띄워서 주소가 이렇게 나와요.

```
https://kakao-impact-foundation.github.io/ax-guides/
```

주소 첫머리에 `kakao-impact-foundation`이 들어가 있으니, 외부에 공유해도 공식 주소처럼 보일 수 있어요. **그래서 재단에서 만든 사이트는 개인 계정이 아니라 재단 Org에 레포를 두고 Github Pages로 배포하는 걸 권장해요.** 개인 계정에 띄워 둔 사이트가 있다면, 2번의 Transfer로 Org에 옮긴 뒤 다시 띄우면 주소가 바뀝니다.

### 띄우는 법 (Github 무료 플랜 기준)

무료 플랜에서는 **공개(Public) 레포만** Pages를 띄울 수 있어요. 그래서 순서는 이래요.

1. 레포를 공개로 전환 (2번의 민감정보 점검을 먼저 거치고)
2. 레포 `Settings` → `Pages` → Branch를 `main` / 폴더 `/`로 지정 → Save
3. 잠시 뒤 `kakao-impact-foundation.github.io/레포이름/` 으로 접속 확인

### ★ 먼저 정하기: 외부 공개용인가, 내부 공유용인가

같은 GitHub Pages라도 **이 사이트의 공개 범위**에 따라 적용해야할 사항이 달라지니, 아래 내용을 확인해 주세요.

| 종류 | 누구에게 보여줄 건가 | 검색 노출 | 접근 제한 | 재단 예시 |
| :-- | :-- | :-- | :-- | :-- |
| **외부 공개용** | 재단 밖 누구나 — **알려지는 게 목적** | 검색 잘 되게 (`noindex` ✗) | 없음 (다 보여주는 게 목적) | 테크포임팩트 캠퍼스 결과 공유 사이트 |
| **내부 공유용** | 내부·관계자만 | **색인 차단(`noindex`)** | **클로드 아티팩트·비번 게이트·서버 검증** | AI 인터뷰어, SNS 콘텐츠 초안봇 |

> 헷갈리면 이렇게 자문해보세요. **"이 주소가 구글 검색에 떠도 괜찮은가?"** 괜찮으면(오히려 떠야 하면) 외부 공개용, 곤란하면 내부 공유용이에요. 색인 차단과 비번 게이트는 **내부 공유용에만** 해당해요.

### ★ 질문 하나 더: 소스 코드(레포)도 공개해도 되나

여기서 많이들 헷갈려요. **사이트(화면)를 누가 보나**와 **소스 코드(레포)를 누가 보나**는 **별개의 질문**이에요. "외부에 보여줘야 하니까 당연히 공개 레포겠지"가 아니라, 둘을 따로 정해야 해요.

- **사이트 공개 여부** = 화면(결과물)을 외부에 보여줄 건가
- **레포 공개 여부** = 소스 코드를 외부에 열어도 되나 (로직·구조가 다 보여요)

이 둘을 조합하면 **어디에 띄울지(플랫폼)** 가 정해져요.

| | 사이트 = 외부 공개 | 사이트 = 내부만 |
| :-- | :-- | :-- |
| **레포 공개 OK** | **GitHub Pages** (무료, 가장 간단) | 권장 ✗ — 소스 열면 비번 보여 차단이 약함. 내부용이면 레포도 비공개로 (아래 칸) |
| **레포는 비공개** | **Vercel Pro** (테일러 문의) | **Vercel 서버 검증** |

> 정적 문서(가이드·리포트·안내)를 내부에만 공유할 거면 위 표처럼 레포·배포를 고민할 필요 없이 **클로드 아티팩트로 발행**하는 게 제일 쉬워요 (→ 5번).

> **레포 비공개 + 외부 공개**는 무료로 안 돼요. GitHub Pages 무료 플랜은 공개 레포만 띄우고, **Vercel 무료(Hobby) 플랜은 Organization 소유 레포 배포가 막혀 있거든요.** 우리 레포는 전부 재단 Org(`kakao-impact-foundation`) 소속이라, 비공개로 두고 외부에 띄우려면 **Vercel Pro**가 필요해요. 테일러가 계정을 갖고 있으니 **테크임팩트팀(테일러)에 문의**하면 연결해드려요.

**재단 실제 예시**

- `techforimpact-ax-interview` — 외부에 나가야 하는 사이트지만 소스는 가려야 해서 **레포 비공개 + Vercel Pro**로 연결돼 있어요.

### 종류와 상관없이 — 공개 레포라 항상 지킬 것

Pages로 띄우려면 레포가 공개여야 하니, **소스 코드가 전부 외부에 보입니다.** 외부 공개용이든 내부 공유용이든 아래는 똑같이 지켜주세요.

- **API 키·비밀번호·토큰을 코드에 넣지 마세요.** 공개 레포에선 그대로 노출돼요. (→ 바로 아래 다루는 법)
- **화면에 그려지는 데이터도 공개**입니다. 개인정보·내부 수치를 화면에 직접 띄우면 안 돼요.
- **AI API(Claude 등)를 쓰면 월 사용 한도·알림을 먼저 거세요.** 비용 폭주를 막는 마지막 방어선이에요 (→ 4-3번).

### 🔑 키·비밀번호는 `.env`로 빼고 레포엔 안 올려요

키를 코드에 직접 적으면 레포가 공개든 비공개든 위험해요 (비공개라도 협업자한테 보이고, 나중에 공개로 바뀔 수도 있으니까). 표준 방법은 이래요.

1. 키는 코드가 아니라 **`.env` 파일에** 적어요. 예: `ANTHROPIC_API_KEY=sk-...`
2. `.env`를 **`.gitignore`에 추가**해서 레포에 아예 안 올라가게 해요.
3. 코드에선 **`process.env.ANTHROPIC_API_KEY`** 처럼 읽어요. (값을 코드에 안 박고 환경에서 꺼내 쓰는 방식)
4. **실제 값은 배포 플랫폼의 환경변수 칸**에 넣어요 — Vercel은 `Settings → Environment Variables`, GitHub Actions는 `Secrets`.

이러면 키가 레포에 **한 번도 들어가지 않고**, 배포 환경에서만 주입돼요.

```
# .gitignore 에 한 줄 추가
.env*
```

> 이미 키를 코드에 올렸다가 지웠어도, **커밋 히스토리엔 그대로 남아 있어요.** 한 번 노출된 키는 지운다고 끝이 아니라 **그 키를 폐기(rotate)하고 새로 발급**해야 해요. 그래서 처음부터 안 올리는 게 제일 중요해요.

### 내부 공유용이라면 — 여기에 더해

외부엔 안 보이고 내부에서만 사용하는 사이트라면, 위 공통 항목에 아래 두 가지를 더 겁니다.

- **검색 색인 차단(`noindex` + `robots.txt`)을 거세요.** 주소를 모르면 못 찾게 (→ 4-2번).
- 화면 앞 **비밀번호(클라이언트)는 공개 레포에선 소스를 열면 보여서** "지나가다 못 들어오는" 약한 차단일 뿐이에요. 정적 문서라면 아예 **클로드 아티팩트**로 발행하는 게 제일 깔끔하고, 서버 로직이 필요하면 Pages 말고 **Vercel 서버 검증**을 쓰세요 (→ 5번).

## 4. 안전하게 배포하기

3번에서 정한 사이트 종류에 따라 챙길 게 달라요.

- **외부 공개용**이면 접근 제한(4-1)·색인 차단(4-2)은 필요 없지만 화면에 민감정보가 없는지만 확실히 확인이 필요합니다.
- **내부 공유용**이면 4-1·4-2를 둘 다 챙겨서 보호 조치합니다.
- **4-3 비용 통제**는 종류와 무관하게 **AI API를 쓰면 항상**이에요.

### 4-1. 접근 제한: 누가 볼 수 있나

가려야 할 데이터가 있느냐에 따라 보호 강도가 달라요.

| 레벨 | 방식 | 강도 | 쓸 곳 |
| :-- | :-- | :-- | :-- |
| 캐주얼 차단 | 화면 앞 비밀번호(클라이언트) + 검색 차단 | 약함  | 민감하진 않지만 검색에 안 걸리게 |
| 실질 인증 | 서버에서 검증(Vercel + 쿠키 + 환경변수) | 강함 | 내부 데이터·업무 도구 |

### 4-2. 검색 색인 차단: 내부 공유용이면

검색엔진에 안 걸리게 막습니다. **내부 공유용 사이트에만** 거세요 — 외부 공개용엔 오히려 검색이 잘 돼야 하니 걸지 않습니다. 정적 사이트면 `<head>`에 아래 한 줄을 넣어주세요.

```html
<meta name="robots" content="noindex, nofollow">
```

`robots.txt`로 `Disallow: /`를, 서버가 있으면 `X-Robots-Tag: noindex` 헤더도 함께 거는 게 좋아요.

### 4-3. API 비용 통제: AI API를 쓰는 도구라면 필수

Claude 같은 API를 쓰는 도구는 잘못하면 비용이 폭주할 수 있어요. **관리자가 콘솔에서 월 사용 한도와 알림을 먼저 걸어두는 게 마지막 방어선**입니다. 앱에서는 요청 횟수 제한(rate limit), 프롬프트 캐싱, 가벼운 작업은 더 저렴한 모델로 분리하는 식으로 아낍니다.

> 비용 통제는 테크임팩트팀이 함께 설정해요. AI API를 쓰는 도구를 만들면 배포 전에 꼭 알려주세요.

## 5. 어디에 올릴까요: 플랫폼 선택

| 상황 | 권장 | 비용 |
| :-- | :-- | :-- |
| 정적 사이트 (소스 공개 OK·외부 공개) | **재단 Org 공개 레포 + GitHub Pages** | 무료 |
| **소스는 가리고(레포 비공개) 외부엔 공개** | **Vercel Pro** (테일러 문의) | 유료 (테일러 계정 보유) ||
| **내부 전용 문서(가이드·리포트·안내)** | **클로드 아티팩트로 발행** (재단 claude.ai 조직만) | 무료 (좌석에 포함) |

> **💡 "외부엔 안 보이고 내부 사람들만" 보게 하려면 — 먼저 클로드 아티팩트를 권해요.**
> 가이드·리포트·안내처럼 화면이 정적인 문서라면, 코드를 **클로드 아티팩트로 발행**하면 끝이에요. 레포·배포 셋업 없이 바로 뜨고, 재단 claude.ai 조직 멤버만 보입니다(외부 공개 설정 자체가 없어서 "지나가다 못 들어오는" 약한 차단이 아니라 조직 로그인 기반의 실질 차단이에요). 크루는 모두 좌석이 있으니 로그인만 하면 바로 봐요. 발행할 때마다 버전으로 쌓이고, 뷰어는 보기만 가능해요.
> 단 아티팩트는 **정적 화면 전용**이라는 걸 기억하세요. 외부 CDN·폰트·이미지는 파일 안에 인라인해야 하고, 서버·API·DB가 붙는 진짜 앱은 안 돼요. 주소도 claude.ai 도메인이라 재단 공식 도메인은 아니고요.
> 꼭 **사이트 형태로 올려야 하거나** 서버 로직이 필요하면, GitHub Pages로 띄우고 **비밀번호 게이트**를 거세요(공개 레포라 클라이언트 비번은 약한 차단이란 점은 감안하고요).

> 외부 공개 정적 사이트는 GitHub Pages(재단 Org 공개 레포), 소스 가리고 외부 공개는 Vercel Pro, 내부 전용 문서는 클로드 아티팩트 — 이 세 가지로 거의 다 해결돼요.

## 6. 배포 전 체크리스트

- [ ] 레포에 API 키·비밀번호가 들어가 있지 않나요 (`.env`, `*.key` 등은 `.gitignore`)
- [ ] 이 사이트가 외부 공개용인지 내부 공유용인지 정했나요 (→ 3번)
- [ ] 내부 공유용이면 접근 제한(서버 비밀번호 게이트)이 실제로 동작하나요
- [ ] 내부 공유용이면 검색 색인 차단(`noindex`)을 걸었나요 (외부 공개용은 반대로 검색이 잘 잡히는지 확인)
- [ ] AI API를 쓴다면 월 한도·알림·요청 제한을 걸었나요
- [ ] 화면에 개인정보·내부 데이터가 그대로 노출되지 않나요
- [ ] README에 공개 범위와 민감도를 적어두었나요
- [ ] 배포 후 실제 URL에서 차단·동작을 눈으로 확인했나요

## 7. 만든 에이전트·스킬은 ax-harness 한 곳에 모아요

각자 만든 Claude 에이전트·스킬은 흩어 두지 말고 공유 레포 한 곳에 모읍니다.
→ **`kakao-impact-foundation/ax-harness`** (조직 안 비공개 레포)

### 어디에 올리나요 (세 칸 중 하나)

| 칸 | 올릴 것 | 경로 |
| :-- | :-- | :-- |
| `shared/` | 누구나 그대로 가져다 쓸 수 있는 **범용** 스킬·에이전트 | `ax-harness/shared/` |
| `team/{팀}/` | 우리 팀이 함께 쓰는 하네스 묶음 | `ax-harness/team/{팀이름}/` |
| `krew/{이름}/` | 내 개인 갤러리 (남이 보고 배우는 용) | `ax-harness/krew/{내영문이름}/` |

지금 `shared`에 스킬 두 개가 올라와 있어요. 누구나 재사용할 만한 범용 도구면 shared 경로에 맞춰 올려주세요. → [shared 폴더](https://github.com/kakao-impact-foundation/ax-harness/tree/main/shared)

### 올리는 법

- GitHub 웹에서 해당 폴더를 열고 `Add file` → `Upload files`로 바로 올려도 돼요 (터미널 없이).
- 올리면 GitHub Actions가 README의 카탈로그를 **자동으로 갱신**해요.
- 각 파일 맨 위 frontmatter(`name`·`description`·`author` 등)만 채우면 카탈로그에 깔끔하게 잡혀요.
- 공유 전에 원온원·예산·개인정보 같은 내부 맥락은 빼주세요. Org 멤버 전원이 보니까요.

## 부록: 관리자(Org Owner)용

- **보안 기본값** (멤버 초대 전 잠금, ✅ 적용 완료): 2FA 강제 · 기본 권한 None/Read · 신규 레포 Private only · Owner 2명 이상(Bus factor 대응)
